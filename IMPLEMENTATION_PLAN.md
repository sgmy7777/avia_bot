# План реализации Telegram-бота для мониторинга авиаинцидентов

## 1) Цели и ожидаемый результат
- Автоматически мониторить новые записи на [aviation-safety.net](https://aviation-safety.net/).
- Обнаруживать только новые инциденты (без дублей).
- Формировать публикацию для Telegram-канала `@avia_crash` по заданному редакционному шаблону.
- Пропускать данные через DeepSeek для рерайта в нейтральном журналистском стиле.
- Публиковать пост в Telegram и логировать результат.

## 2) Архитектура решения
- **Collector (парсер источника):** получает список последних событий с aviation-safety.net.
- **Normalizer:** приводит сырые данные к единому формату (дата, место, ВС, оператор, статус, источник).
- **Deduplicator/Storage:** хранит идентификаторы уже опубликованных событий (SQLite/PostgreSQL).
- **AI Rewriter (DeepSeek):** генерирует готовый текст поста по строгому промпту и валидации.
- **Publisher (Telegram Bot API):** публикует в `@avia_crash`.
- **Scheduler/Worker:** запускает цикл проверки каждые N минут.
- **Observability:** логи, метрики, алерты.

## 3) Технологический стек (рекомендация)
- **Язык:** Python 3.11+
- **HTTP:** `httpx`
- **Парсинг HTML:** `BeautifulSoup4` + `lxml`
- **БД:** SQLite (старт) / PostgreSQL (прод)
- **Планировщик:** APScheduler или Celery Beat
- **Telegram:** `aiogram` или прямой Telegram Bot API
- **LLM API:** DeepSeek Chat/Reasoner через официальный HTTP API
- **Контейнеризация:** Docker + docker-compose

## 4) Поток данных (end-to-end)
1. Scheduler запускает задачу `check_new_incidents`.
2. Collector получает страницу/ленты и извлекает карточки событий.
3. Normalizer строит `incident_id` (например, hash от date+aircraft+location+source_url).
4. Deduplicator сверяет `incident_id` с БД.
5. Для новых инцидентов:
   - собирать факты из первоисточника;
   - формировать `structured_fact_sheet`;
   - отправлять в DeepSeek с системным промптом и правилами;
   - валидировать текст по чек-листу.
6. Publisher публикует сообщение в Telegram-канал.
7. В БД сохраняется статус публикации, текст и timestamp.
8. Ошибки отправляются в служебный чат/лог.

## 6) Промптинг и правила рерайта (DeepSeek)
### 6.1 System prompt (ядро)
- Роль: опытный авиационный редактор-копирайтер.
- Стиль: нейтральный, профессиональный, без сенсационности.
- Ограничение: только проверяемые факты из входных данных.
- Запрет: не делать выводов о причинах без официальных данных.

### 6.2 User prompt (шаблон)
- Передавать структуру фактов: `what/when/where/who/details/injuries/official_statements/investigation/source_links`.
- Требовать формат Telegram:
  - заголовок до 60 символов;
  - 3 абзаца;
  - эмодзи ✈️🚨⚠️;
  - хештеги: `#авиация #происшествие #небонаграни #[регион]` + релевантные.
- Длина: 150–300 слов.

### 6.3 Пост-валидация перед публикацией
- Проверка длины текста.
- Наличие блоков: заголовок, лид, подробности, пострадавшие/расследование.
- Отсутствие спекуляций (эвристика по стоп-словам и формулировкам).
- Наличие ссылки на источник(и).
- Если валидация не прошла — fallback: шаблонная публикация без AI.

## 7) Антидублирование и идемпотентность
- Хранить `source_url`, `incident_id`, `first_seen_at`, `published_at`.
- Уникальный индекс по `incident_id`.
- Повторная попытка публикации только для `status=failed` с ограничением retry.

## 8) Публикация в Telegram
- Бот должен быть администратором канала `@avia_crash` с правом публикации.
- Использовать `sendMessage` с `parse_mode=Markdown` или `HTML`.
- Лимит длины Telegram учитывать заранее.
- При ошибке API: лог + retry с backoff.

## 9) Конфигурация и секреты
Переменные окружения:
- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_CHANNEL=@avia_crash`
- `DEEPSEEK_API_KEY`
- `DEEPSEEK_MODEL`
- `POLL_INTERVAL_MINUTES=10`
- `DATABASE_URL`
- `LOG_LEVEL`

Секреты хранить только в `.env`/vault, не в репозитории.

## 10) Надежность, мониторинг, алерты
- Структурные JSON-логи по каждому шагу.
- Метрики: `incidents_found`, `incidents_published`, `publish_failures`, `rewrite_failures`.
- Алерт в служебный Telegram-чат при серии ошибок.

## 11) Юридические и этические аспекты
- Проверить допустимость парсинга aviation-safety.net (robots.txt, ToS, частота запросов).
- Указывать источник в каждом посте.
- Строго соблюдать правила об освещении жертв и непроверенной информации.

## 12) План реализации по этапам
### Этап 1. MVP (1–2 дня)
- Парсер последних инцидентов.
- SQLite + дедуп.
- Простая Telegram-публикация без AI.

### Этап 2. AI-рерайт (1 день)
- Интеграция DeepSeek.
- Промпт + валидация + fallback.

### Этап 3. Промышленная эксплуатация (1–2 дня)
- Docker, healthchecks, ретраи, алерты, метрики.
- Регрессионные тесты.

## 13) Тестирование
- Unit: normalizer, dedupe, validator.
- Integration: парсинг реальной HTML-страницы (фиксированная копия).
- E2E: новый инцидент -> DeepSeek -> Telegram mock.
- Dry-run режим: формировать посты без публикации.

## 14) Definition of Done
- Новые записи стабильно детектируются.
- Дубли не публикуются.
- Посты соответствуют шаблону и чек-листу.
- Ошибки наблюдаемы, есть автоповторы.
- Канал `@avia_crash` получает корректные публикации.

## 15) Что нужно от вас для старта
1. `TELEGRAM_BOT_TOKEN` и доступ бота в канал `@avia_crash`.
2. API-ключ DeepSeek.
3. Предпочтительный интервал проверки (например, 5/10/15 минут).
4. Подтверждение формата источника (какая именно страница ASN мониторится в приоритете).
